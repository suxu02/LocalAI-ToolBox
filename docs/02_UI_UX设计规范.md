# 🎨 模块1：设计总则与顶层定位

## 1. 设计使命与愿景（Why - 为什么设计）

**核心使命**：**“去工具化”体验设计**。
我们的目标不是打造一个“复杂的软件”，而是提供一个像“纸笔”一样自然顺畅的办公助手。设计必须消除用户对AI技术的畏难情绪，将复杂的底层技术（RAG、LLM）转化为简单直观的“填空”和“提问”交互。

**愿景**：
用户打开产品的第一感觉是——**“这是一个懂我工作的专业助手”**，而不是“一个需要学习的复杂软件”。

---

## 2. 目标用户画像与设计映射（Who - 为谁设计）

基于核心用户“行政/HR”的职业特征，我们将用户画像转化为具体的设计映射：

| 用户特征 | 设计映射（设计策略） |
| :--- | :--- |
| **严谨性要求高** | **视觉风格克制**：严禁使用高饱和度、卡通化、娱乐化的元素。采用“严谨商务风”，色彩以中性色为主，主色点缀，传递稳重、可信赖的感觉。 |
| **文书工作量大** | **排版清晰优先**：内容区（AI回答、文档预览）必须宽敞、易读。字体选用系统默认字体，行间距适中，降低长时间阅读的疲劳感。 |
| **对新技术敏感** | **认知减负**：避免出现“Temperature”、“RAG”、“Embedding”等术语。界面文案必须“说人话”，如用“创造性”代替“Temperature”，用“文档库”代替“向量数据库”。 |
| **容错率低** | **安全感设计**：操作必须可逆（如二次确认删除），数据状态必须可见（如自动保存提示），让用户时刻感到“一切尽在掌握”。 |

---

## 3. 核心设计原则（Principles - 遵循什么）

本产品的设计遵循以下四大铁律，作为设计验收的一票否决项：

### 原则一：效率至上（Efficiency First）
*   **定义**：减少达成目标所需的点击次数和认知成本。
*   **执行标准**：
    *   常用功能（如新建模板、上传文档）必须在**一级页面**可见，不得藏在二级菜单。
    *   AI 生成的过程必须**流式输出**，让用户看到进度，而不是傻等到底。
    *   历史记录必须**实时自动保存**，禁止出现“手动保存”按钮增加用户记忆负担。

### 原则二：所见即所得（WYSIWYG）
*   **定义**：界面呈现的状态应直接反映数据的真实状态。
*   **执行标准**：
    *   **文档解析预览**：用户上传文档后，必须先看到提取的文本（所见），确认无误后才入库（所得），杜绝“黑盒处理”带来的不信任感。
    *   **Markdown 实时渲染**：AI 回答的内容直接渲染为排版好的公文格式，而非让用户看到源码再点击“预览”。

### 原则三：容错与引导（Tolerance & Guidance）
*   **定义**：预防错误发生，并在错误发生时提供明确的解决路径。
*   **执行标准**：
    *   **空状态引导**：当列表为空时，不得只显示“暂无数据”，必须提供引导插图和“立即添加”按钮，告诉用户下一步该做什么。
    *   **错误恢复**：API 报错时，不得只提示错误码，必须提供中文解释和操作建议（如“请检查 Key 是否正确”）。

### 原则四：克制的视觉（Visual Restraint）
*   **定义**：视觉服务于内容，不喧宾夺主。
*   **执行标准**：
    *   **色彩克制**：全站除主色（蓝）、功能色（成功/警告/错误）外，严禁滥用彩色。背景色以白/灰为主。
    *   **装饰克制**：去除不必要的装饰线条、渐变、阴影。保持界面平整、干净，符合“严谨商务风”。

---

## 4. 设计框架顶层架构

为了落实以上原则，我们确立了“分栏式工作台”的顶层架构：

*   **导航层（左侧）**：稳定不动的“地图”。始终展示功能入口和历史记录，让用户清楚“我在哪”、“我去过哪”。
*   **工作层（中间）**：核心作业区。根据功能不同动态切换，但布局逻辑保持一致（上方操作，下方结果）。
*   **辅助层（右侧）**：按需出现的“副驾驶”。仅在需要查看原文、调整参数时展开，平时可收起，避免分散注意力。

# 🎨 模块2：全局设计原子系统（UI 视觉底层基石）

## 1. 色彩系统（Color System）

色彩是视觉层级和状态反馈的核心。我们采用**“克制、清晰”**的配色策略。

### 1.1 品牌主色
用于核心交互、强调信息、选中状态。
*   **标准色**：`#409EFF` (Element Default Blue)
*   **浅色**：`#ECF5FF` (用于背景高亮、标签背景)
*   **深色**：`#337ECC` (用于按钮 Hover 状态)
*   **应用规则**：
    *   主按钮背景、链接文字、选中项边框、聚焦输入框边框 -> **标准色**。
    *   引用卡片背景、标签背景 -> **浅色**。

### 1.2 中性色
构建界面骨架和文本层级。**这是“严谨商务风”的关键，大量使用灰色系。**

| 色值 | 变量名 | 用途 | 备注 |
| :--- | :--- | :--- | :--- |
| `#303133` | --el-text-color-primary | 主文本（标题、重要内容） | 最深，保证可读性 |
| `#606266` | --el-text-color-regular | 常规文本（正文、说明） | 最常用的文字颜色 |
| `#909399` | --el-text-color-secondary | 次要文本（占位符、注释、时间） | 层级最低 |
| `#C0C4CC` | --el-text-color-placeholder | 输入框占位符 | 极浅，用于提示 |
| `#DCDFE6` | --el-border-color | 一级边框（输入框、卡片边框） | 分割区域 |
| `#E4E7ED` | --el-border-color-light | 二级边框（分割线） | 视觉更轻 |
| `#F2F6FC` | --el-fill-color-light | 悬停背景（鼠标移入行/按钮） | 交互反馈 |
| `#F5F7FA` | --el-bg-color-page | 页面背景（整体底色） | 淡灰，减少视觉疲劳 |

### 1.3 功能色
用于状态反馈，严禁挪作他用。
*   **成功**：`#67C23A` (成功 Toast、在线状态)
*   **警告**：`#E6A23C` (警告 Toast、空间不足提示)
*   **危险**：`#F56C6C` (错误 Toast、删除按钮、报错红字)
*   **信息**：`#909399` (普通提示)

## 2. 字体系统

遵循**“系统原生、清晰易读”**原则。

### 2.1 字体族
```css
font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, "Helvetica Neue", Arial, "Noto Sans", sans-serif, "Apple Color Emoji", "Segoe UI Emoji";
```
*   **策略**：不加载外部字体文件。利用系统原生字体实现毫秒级渲染，且不同系统下均显示该平台最标准的商务字体（Win: 微软雅黑 / Mac: 苹方）。

### 2.2 字号与行高阶梯
为了保证阅读舒适度，正文行高设定为 **1.5**。

| 类型 | 字号 | 字重 | 行高 | 应用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **大标题 (H1)** | 20px | Bold (600) | 1.4 | 页面标题（如“提示词工坊”） |
| **小标题 (H2)** | 16px | Bold (600) | 1.5 | 卡片标题、模块标题 |
| **正文 (Body)** | 14px | Regular (400) | 1.5 | 正文内容、表单标签、列表项 |
| **辅助文** | 12px | Regular (400) | 1.5 | 说明文字、时间戳、标签 |

---

## 3. 间距系统

采用 **8pt 网格系统**。所有间距必须是 **4px** 或 **8px** 的倍数，拒绝随意数值。

| Token 名称 | 数值 | 应用场景 |
| :--- | :--- | :--- |
| **$spacing-xs** | 4px | 图标与文字间距、紧密元素间距 |
| **$spacing-sm** | 8px | 按钮内间距、表单项垂直间距 |
| **$spacing-md** | 16px | 模块内元素间距、卡片内边距 |
| **$spacing-lg** | 24px | 区域间距、页面内边距 |
| **$spacing-xl** | 32px | 大板块分隔间距 |

**执行红线**：开发时严禁写 `margin: 13px;` 或 `padding: 9px;`，必须从上述 Token 中选取。

---

## 4. 布局与尺寸

### 4.1 圆角
商务风偏向方正，但为了现代感，保留微小的圆角。
*   **小圆角**：`2px` - 用于按钮、标签、输入框。
*   **中圆角**：`4px` - 用于卡片、下拉面板。
*   **大圆角**：`8px` - 用于模态弹窗。

### 4.2 阴影
商务风阴影必须极其克制，避免厚重感。
*   **基础阴影**：`0 2px 12px 0 rgba(0, 0, 0, 0.05)` - 用于卡片悬浮，极淡。
*   **投射阴影**：`0 2px 8px rgba(0, 0, 0, 0.08)` - 用于下拉菜单、弹窗，层级更高。

---

## 5. 图标系统

*   **库**：Element Plus Icons (线性风格)
*   **尺寸**：
    *   小图标：`12px` 或 `14px` (用于标签内的 icon)
    *   标准图标：`16px` (用于菜单、按钮)
    *   大图标：`20px` 或 `24px` (用于空状态插图)
*   **颜色规则**：
    *   默认跟随文字颜色（`#606266`）。
    *   交互图标 Hover 时变为主色（`#409EFF`）。
    *   禁用状态为占位符颜色（`#C0C4CC`）。

# 🎨 模块3：组件库设计规范（落地核心单元）

**核心策略**：基于 Element Plus 默认主题，通过 CSS 变量微调，确保风格统一。

## 1. 通用交互规范

所有交互组件必须遵循以下状态流转，确保用户操作有反馈。

*   **常态**：默认样式。
*   **悬停**：颜色加深 10% 或背景变为 `#F2F6FC`。
*   **点击/选中**：主色高亮，边框加粗。
*   **禁用**：透明度降低至 0.5，Cursor 变为 `not-allowed`。
*   **聚焦**：外发光阴影（`box-shadow: 0 0 0 2px rgba(64, 158, 255, 0.2)`），用于键盘操作指引。

---

## 2. 导航类组件

### 2.1 侧边栏菜单
用于左侧的功能导航和历史记录列表。

*   **样式规范**：
    *   背景：`#FFFFFF`（白底，通过右边框 `1px solid #E4E7ED` 与内容区分隔）。
    *   菜单项高度：`56px`（舒适点击区）。
    *   选中态：右侧 3px 主色条 + 文字变为主色 + 背景变为淡蓝 `#ECF5FF`。
*   **代码映射**：`<el-menu :default-active="activeIndex" class="el-menu-vertical">`

### 2.2 面包屑
用于告知用户当前位置（如：设置中心 > API配置）。

*   **样式规范**：
    *   字体：`12px`，颜色 `#909399`。
    *   分隔符：`/` 或 `El-Breadcrumb` 默认图标。
*   **代码映射**：`<el-breadcrumb separator="/">`

---

## 3. 数据录入类组件

这是用户产生数据的核心入口，需重点保证易用性。

### 3.1 按钮
*   **主按钮**：用于核心操作（生成、保存）。
    *   样式：背景 `#409EFF`，文字白色，圆角 `4px`。
    *   代码：`<el-button type="primary">`
*   **次按钮**：用于取消、返回等次要操作。
    *   样式：白底，灰边框 `#DCDFE6`，灰字 `#606266`。
    *   代码：`<el-button>`
*   **文字按钮**：用于不显眼的操作（如“清空缓存”）。
    *   样式：无边框，文字 `#409EFF`。
    *   代码：`<el-button type="text">`
*   **危险按钮**：用于删除操作。
    *   样式：背景 `#F56C6C`。
    *   代码：`<el-button type="danger">`

### 3.2 输入框与文本域
*   **样式规范**：
    *   高度：`32px`（紧凑商务风）。
    *   圆角：`4px`。
    *   占位符：`#C0C4CC`，字体 `12px`。
    *   必填标识：Label 前加红色 `*`。
*   **代码映射**：
    *   单行：`<el-input placeholder="请输入内容"></el-input>`
    *   多行：`<el-input type="textarea" :rows="4"></el-input>`

### 3.3 开关
用于简单的二选一（如“开启自动保存”）。

*   **样式规范**：
    *   开启色：`#409EFF`。
    *   关闭色：`#C0C4CC`。
*   **代码映射**：`<el-switch v-model="value">`

### 3.4 上传组件
用于知识库文档上传。

*   **样式规范**：
    *   拖拽区域：虚线边框 `#DCDFE6`，背景 `#FAFAFA`。
    *   图标：`+` 号或上传图标，颜色 `#909399`。
    *   Hover：边框变为主色 `#409EFF`。
*   **代码映射**：`<el-upload drag action="#">`

---

## 4. 数据展示类组件

### 4.1 卡片
用于承载对话内容、历史记录项。

*   **样式规范**：
    *   背景：`#FFFFFF`。
    *   边框：`1px solid #EBEEF5`（极淡）。
    *   圆角：`4px`。
    *   阴影：无（保持平整），仅在 Hover 时加极淡阴影 `0 2px 12px 0 rgba(0,0,0,0.05)`。
*   **代码映射**：`<el-card shadow="hover">`

### 4.2 标签
用于历史记录分类、模板变量标记。

*   **样式规范**：
    *   高度：`24px`。
    *   圆角：`4px`。
    *   背景：淡蓝 `#ECF5FF`，文字 `#409EFF`（默认）。
    *   移除按钮：Hover 时显示 `x`。
*   **代码映射**：`<el-tag closable>公文</el-tag>`

### 4.3 空状态
用于无数据时的展示。

*   **设计规范**：
    *   必须包含：商务风插画（如文件夹、放大镜）+ 灰色说明文案（`#909399`）+ 主操作按钮（主按钮样式）。
    *   **禁止**：仅显示“暂无数据”四个字。

---

## 5. 反馈类组件

### 5.1 消息提示
用于轻量操作反馈。

*   **样式规范**：
    *   位置：页面顶部居中或右上角。
    *   圆角：`4px`。
    *   遮罩：无（不阻断操作）。
    *   图标：成功（绿对勾）、错误（红叉）、警告（黄叹号）。
*   **代码映射**：`ElMessage.success('保存成功')`

### 5.2 消息弹窗
用于重要操作确认（如“确认删除？”）。

*   **样式规范**：
    *   宽度：`420px`。
    *   圆角：`8px`。
    *   按钮：右对齐，“取消”（次按钮）+ “确定”（主按钮/危险按钮）。
*   **代码映射**：`ElMessageBox.confirm('确定删除?', '提示')`

### 5.3 抽屉
用于右侧详情面板（查看引用原文、参数设置）。

*   **样式规范**：
    *   方向：从右向左滑出。
    *   宽度：`350px` 或 `40%`。
    *   标题栏：高度 `48px`，底部分割线，左侧标题，右侧关闭图标。
*   **代码映射**：`<el-drawer title="引用原文" :visible.sync="drawer">`

## 6. 业务复合组件（自定义封装）

针对本项目核心业务，需封装特定的复合组件。

### 6.1 AI 对话气泡
*   **结构**：头像（圆形，AI用图标，用户用首字） + 内容区。
*   **样式**：
    *   AI气泡：白底，左对齐。
    *   用户气泡：淡灰底 `#F2F6FC`，右对齐。
    *   内容区：Markdown 渲染样式需统一（标题字号、代码块背景）。

### 6.2 引用卡片
*   **结构**：文档图标 + 文件名 + 引用片段摘要（截取前 50 字）。
*   **样式**：
    *   背景：淡蓝 `#ECF5FF`。
    *   边框：左侧 3px 主色条。
    *   Hover：光标变手型，背景色加深 5%。
*   **交互**：点击触发右侧 Drawer 展示全文。

# 🎨 模块4：全局交互设计规范（UX 全场景落地规则）

## 1. 导航与跳转规范

### 1.1 路由切换逻辑
*   **单页应用（SPA）体验**：所有页面跳转均不刷新浏览器，采用组件切换模式。
*   **状态保持**：
    *   当用户在“提示词工坊”填写了一半内容，误点“设置”又切回来时，**表单内容必须保留**，严禁清空。
    *   实现方式：使用 Vue 的 `<keep-alive>` 组件缓存页面状态。
*   **右侧面板联动**：
    *   当用户从“知识库”跳转到“历史记录”时，右侧展开的“引用原文面板”必须**自动收起**，避免信息错位。

### 1.2 列表与详情交互
*   **历史记录**：
    *   点击左侧列表项 -> 中间区域加载详情。
    *   **切换逻辑**：若当前对话正在进行（AI正在生成），切换到其他记录时，必须**自动停止**当前生成任务，避免后台资源浪费和混淆。

## 2. 表单与录入规范

### 2.1 防丢失机制（核心体验）
*   **场景**：用户正在编辑模板，未保存就点击了其他菜单。
*   **交互规范**：
    *   系统检测到表单有未保存的变更 -> 阻止跳转 -> 弹出确认弹窗：“当前内容未保存，离开将丢失数据，是否继续？”
    *   按钮：“取消”（留在当前页） / “放弃并离开”。
    *   **严禁**：静默跳转导致用户心血白费。

### 2.2 校验反馈时机
*   **实时校验**：输入框失焦（Blur）时立即触发校验（如格式不对、必填为空）。
*   **提交校验**：点击“生成/保存”时，再次全局校验，若存在错误，自动滚动到第一个报错项，并聚焦输入框。

### 2.3 智能聚焦
*   打开“新建模板”弹窗 -> 自动聚焦到“标题”输入框。
*   键盘 `Tab` 键顺序必须符合视觉顺序（从左到右，从上到下），不得乱跳。

---

## 3. 数据加载与等待规范

### 3.1 加载状态区分
*   **页面初始化**（如打开历史记录）：使用 **骨架屏 (Skeleton)**，占据内容位置，避免布局抖动。
*   **操作反馈**（如保存、上传）：按钮内部显示 **Loading 转圈**，并置灰按钮防止重复点击。
*   **AI 生成过程**：
    *   输入框下方出现 **“思考中...”** 的打字机动画。
    *   流式输出时，输入框自动锁定，下方出现 **“停止生成”** 按钮（红色图标）。

### 3.2 防抖与节流
*   **搜索框**：输入内容时，停止输入 **500ms** 后才发起搜索请求，避免每打一个字就刷一次列表。
*   **保存按钮**：点击后立即禁用，待接口返回成功后再启用，防止重复提交。

# 🎨 模块6：多端适配与响应式设计规范

## 1. 适配策略定义

*   **核心策略**：**PC优先，弹性适配**。
*   **适配范围**：
    *   **核心适配区**：`1366px` 至 `1920px`（主流办公笔记本及台式机分辨率）。
    *   **扩展适配区**：`> 1920px`（大屏显示器，内容区最大宽度限制）。
    *   **降级兼容区**：`< 1366px`（平板/小屏笔记本，提示横向滚动或警告）。
    *   **非适配区**：手机竖屏（`< 768px`）。不重构布局，仅保证页面可滚动查看，并提示“请在电脑端使用”。

---

## 2. 断点与布局规则

利用 CSS 媒体查询和 Element Plus 的栅格系统，定义以下断点规则：

### 2.1 标准桌面端 (Target: 1366px - 1920px)
这是体验最优的主战场。

| 组件 | 布局规则 | 响应式行为 |
| :--- | :--- | :--- |
| **左侧导航栏** | 固定宽度 `220px`。 | 始终可见，不可收起，保证导航效率。 |
| **右侧详情栏** | 默认宽度 `350px`。 | 若内容区空间不足，自动压缩至 `280px`，或通过按钮手动收起。 |
| **中间内容区** | 弹性宽度 `calc(100% - 220px - 350px)`。 | 随窗口大小自由伸缩。内部卡片使用 `%` 宽度。 |

### 2.2 大屏显示器 (Target: > 1920px)
避免内容被过度拉伸导致视线跨度过大。

*   **中间内容区最大宽度限制**：设定 `max-width: 1400px`。
*   **对齐方式**：居中对齐。
*   **留白处理**：左右两侧留出白边，背景色使用 `#F5F7FA`，保持视觉聚焦。

### 2.3 小屏/低分辨率 (Target: < 1366px)
针对行政/HR可能使用的旧款笔记本或缩小浏览器窗口的情况。

*   **警告机制**：当检测到视窗宽度 `< 1300px` 时，界面顶部常驻黄条提示：“当前窗口过小，部分功能可能显示不全，建议全屏或使用大屏设备。”
*   **布局保护**：
    *   **左侧导航栏**：**优先级最高，保持 `220px` 不压缩**。
    *   **中间内容区**：**优先级次之**，最小压缩至 `600px`，出现横向滚动条（优于压缩导致内容错乱）。
    *   **右侧详情栏**：**优先级最低**，默认收起，通过点击按钮浮动展开（Drawer模式），不再挤占中间空间。

---

## 3. 缩放与兼容性适配

### 3.1 浏览器缩放 (Browser Zoom)
行政人员常习惯使用 `Ctrl + +/-` 缩放页面。
*   **适配目标**：支持 `90%` 至 `125%` 的缩放比例。
*   **测试标准**：
    *   在 125% 缩放下，核心操作按钮不得被遮挡。
    *   文字不得出现锯齿或重叠。
    *   所有图标使用 SVG 格式，确保缩放清晰度。

### 3.2 操作系统差异
*   **Windows**：默认字体“微软雅黑”，渲染较粗。
    *   *适配*：字体最小不低于 `12px`，否则会模糊。
*   **macOS**：默认字体“苹方”，渲染细腻。
    *   *适配*：利用 `-webkit-font-smoothing: antialiased` 优化字体显示。

## 4. 移动端“拒绝服务”规范

虽然MVP不支持移动端，但必须有体面的“拒绝”姿势，而非乱码。

**当检测到视窗宽度 < 768px 时：**

2.  **引导界面**：
    *   居中展示产品 Logo。
    *   文案：“AI工具箱为桌面级应用，暂不支持移动端访问。”
    *   辅助文案：“请在电脑浏览器中打开，以获得最佳体验。”
3.  **技术实现**：
    ```javascript
    // 伪代码示例
    if (window.innerWidth < 768) {
      document.body.innerHTML = '<div class="mobile-mask">...</div>';
    }
    ```

# 🎨 模块7：内容与文案设计规范（体验细节核心）

## 1. 文案设计核心原则

所有界面文案必须通过以下四项原则检验：

1.  **用户视角**：多说“我能得到什么”，少说“系统做了什么”。
    *   ❌ 系统：配置已保存。
    *   ✅ 用户：**设置成功，配置已生效。**
2.  **精准无歧义**：一个词只代表一个含义。
    *   ❌ “删除”：是指删除这条记录，还是删除整个文档？
    *   ✅ **“删除记录”** vs **“彻底移除文档”**。
3.  **动词驱动**：按钮文案必须是动作，让用户知道下一步做什么。
    *   ❌ “下一步”、“确认”
    *   ✅ **“开始生成”、“立即导入”、“前往充值”**。
4.  **去技术化**：严禁直接展示技术术语，必须进行“人话翻译”。

## 2. 术语定义与统一命名

# 🎨 模块8：设计研发协同与交付规范（落地保障）

## 1. 设计交付标准

本项目的交付物不仅是“图片”，而是**“可直接开发的配置参数”**。

### 1.1 必须交付的要素
每一个功能页面开发前，必须具备以下三要素，否则严禁开工：
1.  **布局参数**：明确标注容器宽度、间距、断点。
2.  **组件映射表**：明确指明 UI 元素对应的 Element Plus 组件名称（如 `el-button`, `el-card`）。
3.  **状态定义**：必须包含 常态、悬停、点击、禁用 四态定义，以及空状态/加载态设计。

### 1.2 切图与资源规范
*   **格式**：仅接受 SVG 格式（图标/插图）。
*   **命名**：全小写，连字符分隔。
    *   格式：`模块名_功能名_状态.svg`
    *   示例：`knowledge_upload_hover.svg`, `icon_logo.svg`

---

## 2. 代码实现与还原规范

### 2.1 样式隔离与变量化
严禁在 `.vue` 文件的 `<style>` 标签内直接写死颜色值（如 `color: #409EFF`）或像素值。
*   **正确做法**：使用 CSS 变量或 SCSS 变量。
    ```scss
    /* 错误写法 */
    .my-button { background: #409EFF; }

    /* 正确写法 */
    .my-button { background: var(--el-color-primary); }
    ```
*   **全局变量文件**：所有设计原子（颜色、字体、间距）统一定义在 `src/assets/styles/variables.scss` 中。

### 2.2 组件库使用红线
*   **禁止自定义造轮子**：凡 Element Plus 有的组件，严禁手写 HTML/CSS 模仿。
    *   例如：不要用 `<div class="btn">` 模仿按钮，必须用 `<el-button>`。
*   **禁止全局覆盖组件库样式**：严禁修改 Element Plus 的全局源码样式。
    *   如需微调，使用 CSS 深度选择器 (`:deep()`) 在当前页面/组件内覆盖。

### 2.3 布局实现标准
*   **严禁使用 `px` 固定布局**（除特定图标外）。
*   主布局必须使用 Flexbox 或 Grid，配合百分比 `%` 或 `flex-grow` 实现弹性伸缩。
*   **验收标准**：拉伸浏览器窗口时，布局不得错位、重叠。

---

## 3. 还原度验收流程

开发完成后，必须执行“自查验收”，流程如下：

1.  **像素级对比**：在 Chrome 开发者工具中，将设计稿（如有）与实现界面重叠或并排对比。
2.  **断点测试**：
    *   调整视窗宽度至 `1366px`（最小适配宽度） -> 检查右侧面板是否按规范变为浮动 Drawer。
    *   调整至 `1920px` -> 检查中间内容区是否居中且有最大宽度限制。
3.  **状态走查**：
    *   依次触发 悬停、点击、禁用、报错 状态，核对视觉反馈是否符合设计规范。
    *   特别检查 **空状态**：列表为空时，是否显示了引导插图而非空白。

## 4. 变更管理规范

需求变更是常态，但必须遵循“同步变更”原则。

### 4.1 变更流程
1.  **源头记录**：任何设计变更（如改颜色、改文案），必须先更新设计规范文档/设计稿，标注修改日期。
2.  **代码同步**：基于更新后的文档修改代码，严禁“改代码不改文档”或“改文档不改代码”。
3.  **影响范围评估**：
    *   若涉及 **原子变量**（如主色值）变更，必须全站排查所有页面。
    *   若涉及 **文案** 变更，需搜索代码中的国际化字段或硬编码字符串。

### 4.2 废弃资源清理
*   变更后删除的图标、图片资源，必须立即从项目中删除，不得留在代码库中“以防万一”。

## 3. 迭代优化机制（持续改进）

产品上线不是结束，而是改进的开始。

### 3.1 问题分级与处理时限

根据问题对用户体验的影响程度，分为三级，设定不同的处理时限：

| 级别 | 定义 | 响应时间 | 解决时限 |
| :--- | :--- | :--- | :--- |
| **P0 (阻断级)** | 核心功能不可用（如API连不上、文档解析崩溃、数据丢失）。 | **立即** | **< 24小时** (热修复) |
| **P1 (体验级)** | 功能可用但体验极差（如报错看不懂、布局错位、操作繁琐）。 | **< 12小时** | **< 3天** (随下个版本发布) |
| **P2 (优化级)** | 视觉瑕疵、文案优化、非核心功能的交互细节。 | **< 3天** | **< 1周** (积累后统一优化) |

### 3.2 体验迭代流程
1.  **收集**：记录自测发现的问题 + 用户反馈。
2.  **定级**：按 P0/P1/P2 分类。
3.  **排期**：P0 立即修；P1/P2 放入 Backlog（待办列表），每周五进行一次清理。
4.  **验证**：修复后，必须重新执行对应的“验收标准”。

---

## 4. 版本迭代规范

遵循**语义化版本控制 (SemVer)**，但在 MVP 阶段允许快速迭代。

*   **版本号格式**：`Major.Minor.Patch` (如 `1.0.1`)
    *   **Major (主版本)**：大规模重构、架构调整、不兼容的更新。（如：增加用户系统）
    *   **Minor (次版本)**：新增功能、显著的体验优化。（如：新增“导出Word”功能）
    *   **Patch (补丁版)**：Bug修复、文案调整、微小的界面微调。（如：修复PDF解析乱码）

**迭代节奏**：
*   **MVP上线后第1个月**：每两周一个 Minor 版本（快速响应用户反馈）。
*   **稳定期后**：每月一个 Minor 版本。
