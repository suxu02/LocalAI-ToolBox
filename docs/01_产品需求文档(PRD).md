# 📦 模块1：项目概述与核心基线

## 1. 项目背景与愿景（Why - 为什么做）

### 1.1 核心痛点
在AI普及的时代，行政与HR白领面临着**“最后一公里”的效率困境**：
*   **重复劳动**：每次使用AI都要重新输入冗长的背景信息和指令。
*   **上下文缺失**：通用AI助手不了解公司具体的制度、流程和过往案例，生成的答案往往“正确但无用”。
*   **隐私顾虑**：由于担心公司内部数据泄露，不敢将敏感文档上传至公有云AI平台。

### 1.2 产品愿景
打造一款**“懂你的私有AI副驾驶”**。
它不需要联网注册，打开浏览器就能用；它记住了你的常用指令（模板）和公司制度（知识库）；它把数据完全留在你的电脑里，让你在安全的前提下，实现真正的工作提效。

---

## 2. 目标用户与场景（Who - 为谁做）

### 2.1 核心画像
*   **职业**：行政专员、HR专员、部门助理。
*   **特征**：文书工作量大，重复性高，对数据安全敏感，对技术有畏难情绪。
*   **技术能力**：会使用浏览器，会基本的文档操作，不懂编程。

### 2.2 典型场景
1.  **公文撰写**：需要频繁发布通知、拟定邮件，希望一键生成初稿，而非从零开始。
2.  **制度查询**：面对几百页的员工手册，希望能像对话一样快速找到答案，而不是翻文档。
3.  **经验复用**：上周写过的方案，这周类似的需求希望能直接调出来改改。

## 4. 产品边界与红线（Not - 不做什么）

这是防止项目烂尾和需求蔓延的关键，请务必明确：

### 4.1 明确不做（Won't List）
1.  **不做用户系统**：MVP阶段无注册、登录、密码找回功能。打开即用，数据在本地。
2.  **不做云端同步**：不支持多设备数据自动同步。换设备需手动“导出-导入”数据包。
3.  **不做复杂OCR**：不支持扫描件PDF、图片文字识别。仅支持可复制的文字版文档。
4.  **不做团队协作**：不支持多人共享知识库或模板。
5.  **不做移动端适配**：优先保证PC端浏览器体验，移动端访问可能布局错乱（暂不处理）。

### 4.2 技术约束（Constraints）
1.  **存储限制**：受浏览器IndexedDB限制，单站点建议存储数据不超过500MB。
2.  **网络依赖**：核心生成功能需联网调用DeepSeek API；本地数据管理（模板/历史）可离线操作。

### 4.3 V1.1 架构约束
**【V1.1 新增】**
1.  **数据库兼容性**：
    *   `knowledge` 表新增 `mode` 和 `chunks` 字段，旧数据默认为 `'full'` 模式。
    *   **严禁修改已有字段的数据类型**，确保 V1.0 用户数据平滑升级。
2.  **接口稳定性**：
    *   `useAI.js` 的核心接口保持不变。
    *   新增 BM25 算法模块，不修改原有逻辑。

## 5. 成功基线（Success Baseline）

### 5.1 V1.1 新增验收标准
**【V1.1 新增】**
1.  **新手友好度**：首次打开应用，3 秒内看到欢迎弹窗；未配置 Key 时，设置图标旁有红点提示。
2.  **数据完整性**：V1.0 用户升级到 V1.1 后，所有历史数据完整保留，且默认兼容新模式。

只要满足以下标准，MVP即可视为成功上线：

1.  **可用性**：用户能够在3分钟内完成API配置，并成功生成第一段文本。
2.  **准确性**：知识库问答的回答必须基于上传的文档，并能展示引用来源。
3.  **安全性**：用户清理浏览器缓存前，数据永久保留；导出的备份文件可完整恢复数据。
4.  **容错性**：上传错误格式或API Key填错时，有明确的报错提示，不出现白屏。

# #1：设置中心 & 基础设施

**模块目标**：为系统提供运行底座，确保用户能连通AI，且数据资产绝对安全。

### 1.1 功能点：API Key 配置与管理
*   **功能描述**：用户输入并保存 DeepSeek API Key，系统负责安全存储并提供连通性测试。
*   **操作逻辑**：
    1.  输入框默认为空，placeholder 提示：“请输入 DeepSeek API Key”。
    2.  输入内容默认显示为密文（****），右侧提供“明文/密文切换”图标。
    3.  点击“保存并测试”按钮，系统将 Key 存入 IndexedDB，并发送一个极简请求（如“hi”）验证有效性。
    4.  验证成功弹出“配置成功”；验证失败弹出具体错误（如“余额不足”、“Key格式错误”）。
*   **验收标准**：
    *   **AC-1.1.1**：输入 Key 保存后，刷新页面，Key 值依然存在且处于密文状态。
    *   **AC-1.1.2**：输入错误的 Key（如“123”），点击保存，系统拦截并提示“格式错误”。
    *   **AC-1.1.3**：输入正确 Key 但无余额，系统提示“API余额不足”而非报错代码。

### 1.2 功能点：数据备份与恢复（兜底核心）
*   **功能描述**：支持将本地全量数据导出为 JSON 文件，并支持导入恢复。
*   **操作逻辑**：
    1.  **导出**：点击“导出数据”，系统读取 IndexedDB 所有数据，生成 `local_ai_toolbox_backup_[时间戳].json` 文件并下载。
    2.  **导入**：点击“导入数据”，选择本地 JSON 文件。系统校验文件格式：
        *   若格式正确，提示“将覆盖当前数据，是否继续？”。
        *   若格式错误，提示“非标准备份文件”。
    3.  导入成功后，页面自动刷新，左侧历史列表显示导入的数据。
*   **验收标准**：
    *   **AC-1.2.1**：导出的 JSON 文件包含 `settings`、`templates`、`knowledge`、`history` 四个完整字段。
    *   **AC-1.2.2**：在浏览器清空缓存后，执行导入操作，历史记录和模板列表均能恢复。
    *   **AC-1.2.3**：导入非 JSON 文件或损坏文件，系统捕获异常，不发生页面崩溃。

#### 1.21 功能点：数据备份与恢复（UI 调整）
*   **位置调整**：将导出/导入按钮置于“API 配置”卡片**下方**，形成独立的“数据管理”区域。
*   **UI 规范**：
    *   区域标题：“数据管理”。
    *   操作按钮：“导出数据包”、“导入数据包”。
*   **验收标准**：
    *   **AC-1.2.1**：在设置页能直接看到“数据管理”区域，无需滚动。
    *   **AC-1.2.2**：导出/导入逻辑与 V1.0 保持一致，确保兼容性。

##2：提示词工坊

**模块目标**：将隐性经验显性化，通过模板复用实现“填空即得”。

### 2.1 功能点：模板管理（CRUD）
*   **功能描述**：对提示词模板进行增删改查，支持变量自动识别。
*   **操作逻辑**：
    1.  **新建/编辑**：提供标题输入框和内容编辑区。
    2.  **变量识别**：用户输入 `{{活动主题}}`，系统实时高亮显示变量（如蓝色背景）。
    3.  **高级设置**：集成“创造性滑块”（Temperature 0-1），默认值 0.7。
    4.  **保存**：校验标题不为空，保存至 IndexedDB。
*   **验收标准**：
    *   **AC-2.1.1**：输入包含 `{{}}` 的文本，系统自动识别并生成标签展示在下方（如“检测到变量：活动主题”）。
    *   **AC-2.1.2**：滑块调整到 0.2，保存后重新进入编辑页，滑块位置保持在 0.2。

### 2.2 功能点：填空生成（核心交互）
*   **功能描述**：选中模板后，动态生成表单，用户填空后调用 AI 生成。
*   **操作逻辑**：
    1.  点击模板列表项，中间区域加载“动态表单”。
    2.  表单项根据模板变量自动生成（如模板有 `{{时间}}`，表单则有“时间”输入框）。
    3.  用户点击“生成”，界面显示 Loading 动画，AI 回答区域流式输出文本。
*   **验收标准**：
    *   **AC-2.2.1**：动态表单生成的输入框 Label 与模板变量名完全一致。
    *   **AC-2.2.2**：生成过程中，用户可随时点击“停止生成”中断输出。
    *   **AC-2.2.3**：生成结果必须基于填写的变量内容（如填“年会”，结果必须是关于年会的）。

**【V1.1 新增】预置模板**
*   **功能描述**：系统首次初始化时，自动向数据库写入一个“公文生成器”模板。
*   **模板内容**：
    ```text
    你是一个专业的行政助理。请根据以下信息，撰写一份正式的公文通知。
    要求：
    1. 语气严谨、规范。
    2. 格式标准，包含标题、正文、落款日期。
    3. 核心信息如下：
       - 通知主题：{{通知主题}}
       - 发布单位：{{发布单位}}
       - 核心内容：{{核心内容}}
    ```
*   **验收标准**：
    *   **AC-2.X.1**：全新安装环境下，工坊列表默认有一条“公文生成器”模板。

##3：本地知识库

**模块目标**：实现私有数据的问答，并确保溯源的准确性。

### 3.1 功能点：文档上传与文本处理
*   **功能描述**：上传 PDF/Word，提取文本，支持人工校验。
*   **操作逻辑**：
    1.  用户拖拽或点击上传文件。
    2.  前端解析文件内容，进度条显示“正在解析...”。
    3.  解析完成后，弹出**“文本预览与编辑”弹窗**：
        *   展示提取出的纯文本内容。
        *   用户可编辑修正（如删除乱码）。
    4.  用户点击“确认入库”，系统将纯文本存入 IndexedDB。
*   **验收标准**：
    *   **AC-3.1.1**：上传一个 10 页 PDF，解析时间 < 10 秒，且弹窗展示文本内容。
    *   **AC-3.1.2**：上传加密 PDF 或扫描件图片，系统提示“暂不支持此格式，请检查文件”。
    *   **AC-3.1.3**：在编辑弹窗中删除部分文字后入库，数据库中存储的应为修改后的内容。
**【V1.1 新增】分块逻辑**
*   若识别为长文档（> 5000字），解析后自动触发分块：
        1.  按段落或固定字数（500字）切分。
        2.  构建 BM25 索引（词频统计）。
       3.  存储 `chunks` 数组到数据库。

### 3.2 功能点：知识问答与溯源
*   **功能描述**：基于已入库文档进行对话，并展示引用来源。
*   **操作逻辑**：
    1.  用户选择已入库文档（如《员工手册》）。
    2.  输入问题：“婚假怎么算？”。
    3.  **前端 RAG 逻辑**：
        *   系统在本地文档全文中搜索关键词“婚假”。
        *   截取关键词前后各 500 字作为上下文。
        *   拼接提示词发送给 DeepSeek。
    4.  AI 回答后，系统将刚才截取的片段作为“引用卡片”附在回答下方。
    5.  点击卡片，右侧面板跳转并高亮显示该片段。
*   **验收标准**：
    *   **AC-3.2.1**：回答内容必须包含与问题相关的信息，严禁凭空捏造。
    *   **AC-3.2.2**：引用卡片必须显示来源文档名称（如“来源：《员工手册》”）。
    *   **AC-3.2.3**：点击引用卡片，右侧面板自动滚动到对应位置，且文字高亮持续 3 秒。

**模块目标**：实现“长文档精准检索”与“短文档全文投喂”的双模式架构。

### 3.3 功能点：双模式架构设计
**【V1.1 新增】**
*   **基础模式（全文投喂）**：
    *   适用场景：文档 < 5000 字。
    *   逻辑：保持 V1.0 逻辑不变，将文档全文作为 Context 发送给 AI。
*   **高级模式（智能分块检索）**：
    *   适用场景：文档 > 5000 字，或用户主动选择。
    *   逻辑：
        1.  上传时将文档切分为多个 Chunk（每块约 500 字）。
        2.  用户提问时，使用 BM25 算法检索最相关的 3-5 个 Chunk。
        3.  拼接相关 Chunk 作为 Context 发送给 AI。

### 3.4 功能点：模式自动识别
**【V1.1 新增】**
*   **逻辑**：
    *   系统自动计算文档字数。
    *   < 5000 字：自动标记为 `'full'` 模式。
    *   \> 5000 字：自动标记为 `'chunk'` 模式，并提示“已启用智能分块检索”。
*   **UI 表现**：
    *   文档列表中显示模式标签：“全文模式”或“智能模式”。

### 3.5 功能点：智能检索逻辑
**【V1.1 新增】**
*   **智能模式**：
    *   提问 -> BM25 检索 Top 3 相关 Chunk -> 拼接 Context -> 发送 AI。
    *   **验收标准**：
        *   **AC-3.4.1**：上传一份 10000 字的员工手册，提问具体条款，AI 回答准确，且未消耗过多 Token。

##4：历史档案馆

**模块目标**：确保所有数字资产可追溯、可检索。

### 4.1 功能点：自动保存与列表展示
*   **功能描述**：所有对话自动落盘，支持按时间倒序排列。
*   **操作逻辑**：
    1.  用户发起任何对话，系统自动生成标题（取首句前 10 字）并保存。
    2.  左侧列表实时更新记录。
*   **验收标准**：
    *   **AC-4.1.1**：生成一段对话后，刷新页面，该对话依然存在。
    *   **AC-4.1.2**：列表按时间倒序，最新记录置顶。

### 4.2 功能点：标签与搜索
*   **功能描述**：对历史记录打标签，支持多维度筛选。
*   **操作逻辑**：
    1.  **打标签**：用户在历史详情页，输入“#公文”，按回车确认。
    2.  **筛选**：点击左侧标签树（如“公文”），列表仅显示含该标签的记录。
    3.  **搜索**：在搜索框输入“团建”，列表实时模糊匹配标题或内容。
*   **验收标准**：
    *   **AC-4.2.1**：同一条记录可打多个标签（如“#公文 #已发送”）。
    *   **AC-4.2.2**：搜索“团建”，能匹配到标题含“团建”或内容含“团建”的记录。

# 📦 模块3：非功能需求详述（NFR）

**核心原则**：在普通办公电脑环境下，保证流畅、稳定、安全，严禁出现“能用但难用”的情况。

## 1. 性能效率（Performance）

# 📦 模块5：项目约束与风险管控

## 1. 硬性约束（不可突破的边界）

这些是项目的物理极限，任何需求和技术方案都不得试图挑战这些边界，否则项目必然烂尾。

### 1.1 技术约束
*   **纯静态部署**：项目最终产物必须是一个纯静态文件包（HTML/CSS/JS），可直接双击打开或部署在任意静态托管服务（如 GitHub Pages、Vercel）上，**严禁依赖任何后端服务器（Node/Python/Java等）**。
*   **存储天花板**：浏览器 IndexedDB 单站点存储上限约为 500MB-800MB（视浏览器而定）。
    *   **强制约束**：系统必须在前端限制上传文件总大小，设定**硬阈值上限为 400MB**。一旦超过，系统禁止上传并提示清理，严禁尝试突破浏览器限制。
*   **计算能力**：前端 JS 是单线程，无法处理大规模并发计算。
    *   **强制约束**：文档解析必须采用分片/异步处理，解析超过 20 页的文档时，必须显示进度条，**严禁冻结 UI 线程**导致页面卡死。

### 1.2 资源约束
*   **人力**：仅 1 人（你）。
    *   **强制约束**：任何预估工时超过 **3 天** 的单功能必须拆解或砍掉；任何涉及多人协作流程的功能（如审批流、共享编辑）一律禁止。
*   **时间**：MVP 目标周期 4 周（假设每天 2-3 小时）。
    *   **强制约束**：严格执行“核心功能优先”原则。若第 3 周核心功能未完成，必须砍掉“历史标签”、“参数预设”等非核心功能，保核心上线。

### 1.3 外部依赖约束
*   **API 依赖**：DeepSeek API。
    *   **强制约束**：必须提供“自定义 API Base URL”输入框。一旦 DeepSeek 官方 API 域名被墙或服务中断，用户可自行配置代理地址，确保工具不瘫痪。

## 2. 核心风险识别与应急预案

基于 100+ 项目实战经验，我为你预判了本项目最大的 3 个风险点，并制定了强制执行的应急预案。

### 风险一：用户数据丢失（清理缓存/浏览器重装）
*   **风险等级**：🔴 高危（直接导致用户流失）
*   **根源**：纯本地存储天然脆弱，用户随手清理浏览器缓存，数据就没了。
*   **应急预案（强制执行）**：
    1.  **首次启动引导**：用户首次打开工具，必须弹出强提示引导：“本工具数据仅存于本地，请务必定期使用【导出备份】功能”。
    2.  **本地备份检测**：如果检测到用户连续 7 天未进行“导出备份”操作，界面顶部常驻黄条提醒：“为了数据安全，建议立即备份数据”。
    3.  **自动保存机制**：所有操作实时落盘（IndexedDB），避免仅在内存中操作。

### 风险二：API Key 泄露或被盗刷
*   **风险等级**：🟡 中危
*   **根源**：用户在公共电脑使用，或误将包含 Key 的截图发给别人。
*   **应急预案（强制执行）**：
    1.  **显隐切换**：API Key 输入框默认密文，且无“复制”按钮，增加盗取难度。
    2.  **使用提醒**：在设置页显著位置标注：“请勿将 API Key 分享给他人，这可能导致您的余额被盗刷”。
    3.  **前端限制**：虽然前端无法完全防止盗刷（代码可见），但可以通过简单的混淆或运行时注入，增加直接从代码中提取 Key 的难度。

### 风险三：文档解析失败率过高（乱码/格式错误）
*   **风险等级**：🟡 中危
*   **根源**：前端解析库（pdf.js / mammoth.js）对复杂排版（表格、分栏、图文混排）解析能力有限。
*   **应急预案（强制执行）**：
    1.  **预期管理**：在知识库上传页显著标注：“建议上传纯文字版 PDF，表格/图片可能导致解析错误”。
    2.  **人工兜底**：**坚持执行“预览编辑”流程**。解析后允许用户手动修正乱码，这是唯一能解决此问题的低成本方案。
    3.  **降级方案**：若解析失败，提示“文档结构复杂，解析失败”，引导用户复制文本直接粘贴到对话框，不走知识库流程。

## 3. 项目熔断机制

一旦出现以下情况，必须立即触发熔断，重新评估项目走向：

1.  **技术熔断**：如果在开发过程中发现 IndexedDB 在某些浏览器版本下存在严重的 Bug 导致数据无法写入，且无前端解决方案。
    *   *对策*：立即暂停开发，方案调整为“Electron 桌面应用”（开发成本增加）或“放弃离线功能，引入轻量后端”。
2.  **成本熔断**：如果发现 DeepSeek API 价格暴涨，或停止对新用户注册。
    *   *对策*：立即适配其他模型（如 OpenAI / 文心一言），MVP 阶段必须保证多模型接口兼容性。
3.  **时间熔断**：如果在第 2 周结束时，核心的“知识库问答”功能仍无法跑通。
    *   *对策*：砍掉“知识库”模块，仅保留“提示词工坊”作为第一版产品先行上线，验证“模板提效”的价值。

1.  **《用户操作手册》**（1页纸）：简要说明如何配置 Key、如何导出备份。
2.  **《README.md》**：项目说明文件，包含如何本地运行、技术栈说明。
3.  **《更新日志》**：记录 V1.0 包含的功能点。
